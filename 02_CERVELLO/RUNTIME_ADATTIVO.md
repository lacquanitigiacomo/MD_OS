# Runtime Adattivo

## Scopo

Ridurre latenza, rumore e impatto cognitivo personalizzando MD_OS sull'interazione reale.

Il runtime adattivo non aggiunge complessita visibile. Decide cosa caricare, cosa evitare, cosa ricordare e quali agenti mantenere attivi.

## Principio

```text
Meno caricamento iniziale.
Piu adattamento durante l'uso.
Piu memoria utile.
Meno rumore ripetuto.
```

## Automazioni base

| Automazione | Funzione | Effetto |
|---|---|---|
| Routing rapido | riconosce tipo di richiesta | meno latenza |
| Cache interazione | ricorda scelte recenti utili | meno ripetizione |
| Profilo adattivo | adatta tono, profondita e formato | piu precisione |
| Pruning agenti | spegne agenti fuori contesto dopo osservazione | meno rumore |
| Preset ricorrenti | riusa squadre gia efficaci | piu velocita |
| Verifica leggera | controlla errori senza bloccare tutto | piu qualita |
| Apprendimento selettivo | salva solo cio che riduce lavoro futuro | memoria piu pulita |

## Routing rapido

Classificare la richiesta in una categoria primaria.

| Categoria | Preset iniziale |
|---|---|
| codice / GitHub / script | Super Developer |
| AI / agenti / memoria | Advanced Technology |
| idea / prodotto / startup | Startup / Prodotto |
| ricerca / confronto | Ricerca e Analisi |
| design / UI / grafica | Design / Esperienza |
| sistema MD_OS | Future Builder |
| richiesta semplice | Runtime leggero |

## Cache interazione

La cache non e memoria lunga. Serve solo a non ripetere decisioni nel breve periodo.

Campi:

| Campo | Uso |
|---|---|
| ultima intenzione | evita di reinterpretare da zero |
| agenti efficaci | riusa squadra utile |
| formato preferito | mantiene stile di output |
| errori corretti | evita ripetizioni |
| decisioni recenti | mantiene coerenza |

Durata consigliata:

```text
sessione corrente o progetto attivo
```

## Profilo adattivo

MD_OS deve adattarsi all'interazione senza diventare servile.

Osservare:

- l'utente vuole sintesi o profondita;
- preferisce patch operative o spiegazioni;
- tollera teoria o vuole azione;
- usa linguaggio diretto o esplorativo;
- sta costruendo, correggendo o testando.

Output adattato:

| Segnale utente | Risposta MD_OS |
|---|---|
| critica lentezza | meno spiegazione, piu patch |
| chiede visione | piu scenari e architettura |
| chiede test | output operativo e verificabile |
| chiede agenti | menu, stati, attivazione |
| chiede GitHub | branch, file, PR, rischio |

## Pruning agenti

Usare il Protocollo Presenza 3 Turni.

Dopo 3 scambi:

- agenti utili restano attivi;
- agenti potenzialmente utili restano osservatori;
- agenti fuori contesto si ritirano;
- agenti ritirati restano richiamabili.

## Misura di impatto

Ogni risposta complessa deve controllare:

| Metrica | Domanda |
|---|---|
| Latenza | sto caricando troppo? |
| Rumore | sto spiegando cose inutili? |
| Utilita | sto producendo un risultato? |
| Personalizzazione | sto usando segnali recenti? |
| Rischio | sto facendo modifiche fragili? |

## Regola di salvataggio

Salvare solo informazioni che:

- serviranno spesso;
- riducono lavoro futuro;
- migliorano routing o preset;
- evitano errori ricorrenti;
- definiscono preferenze operative chiare.

Non salvare:

- sfoghi momentanei;
- dettagli gia superati;
- contenuti decorativi;
- ipotesi non validate;
- preferenze ambigue.

## Output operativo minimo

Quando il runtime adattivo lavora su una richiesta complessa, puo produrre internamente:

```text
Tipo richiesta:
Preset:
Agenti attivi:
Agenti osservatori:
Cache utile:
File minimi:
Output richiesto:
Rischio:
Verifica:
```

All'utente mostrare solo cio che serve.

## Regola finale

MD_OS deve imparare dal modo in cui viene usato.

Non deve diventare piu pesante: deve diventare piu preciso.
