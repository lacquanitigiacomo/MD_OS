# Runtime Base 5/5/5

## Scopo

Definire il nucleo operativo minimo di MD_OS 10G.

Il runtime base non carica tutto il sistema. Carica una selezione abbastanza ricca da lavorare bene, ma abbastanza leggera da non generare rumore.

## Formula

```text
5 professioni
+ 5 skill
+ 5 matrici
= pacchetto runtime base
```

## Regola principale

Usare il runtime base quando la richiesta e generica, progettuale, creativa, tecnica o strategica ma non richiede ancora una specializzazione estrema.

Espandere oltre il 5/5/5 solo quando la richiesta lo giustifica.

## 5 professioni base

| Professione | Funzione | Quando si attiva |
|---|---|---|
| Architetto di sistema | Disegna struttura, moduli, dipendenze e flussi | quando serve organizzare un sistema, un progetto o una procedura |
| Stratega | Decide priorita, sequenze, scenari e traiettorie | quando serve scegliere cosa fare prima e perche |
| Sviluppatore | Costruisce codice, automazioni, integrazioni e strumenti | quando serve produrre qualcosa di tecnico o eseguibile |
| Grafico | Cura leggibilita, interfaccia, markdown, mappe e forma visiva | quando l'output deve essere chiaro, ordinato e presentabile |
| Verificatore | Controlla errori, rischi, coerenza e qualita | sempre, soprattutto prima dell'output finale |

## 5 skill base

| Skill | Funzione | Risultato atteso |
|---|---|---|
| Routing professioni | Decide quali professioni attivare | squadra minima utile |
| Compressione contesto | Riduce rumore e seleziona solo cio che serve | contesto leggero |
| Mappatura concettuale | Collega idee, file, moduli e dipendenze | struttura leggibile |
| Sintesi generativa | Produce output compatti ma completi | risposta utile |
| Verifica avversaria | Stress test su errori, buchi logici e rischi | output piu robusto |

## 5 matrici base

| Matrice | Funzione | Decisione prodotta |
|---|---|---|
| Impatto / Sforzo | Decide cosa fare prima | priorita operativa |
| Segnale / Rumore | Decide cosa tenere e cosa scartare | selezione contesto |
| Profondita / Urgenza | Decide quanto approfondire | livello di dettaglio |
| Rischio / Reversibilita | Decide se procedere, proteggere o rimandare | livello di cautela |
| Qualita Output | Valuta chiarezza, utilita, precisione e completezza | revisione finale |

## Sequenza runtime

```text
1. Capire intenzione reale
2. Attivare 5 professioni base
3. Applicare 5 skill base
4. Valutare con 5 matrici base
5. Generare output
6. Verificare output
7. Salvare apprendimento solo se utile
```

## Criteri di espansione

Aggiungere professioni, skill o matrici extra solo se almeno una condizione e vera:

- la richiesta appartiene a un dominio specialistico;
- l'utente chiede esplicitamente piu profondita;
- il rischio di errore e alto;
- l'output deve diventare un file, un piano o un sistema stabile;
- il runtime base non basta a decidere bene.

## Esempio runtime

Richiesta:
> Popola i file di MD_OS senza appesantire il sistema.

Professioni attive:
- Architetto di sistema;
- Stratega;
- Sviluppatore;
- Grafico;
- Verificatore.

Skill attive:
- Routing professioni;
- Compressione contesto;
- Mappatura concettuale;
- Sintesi generativa;
- Verifica avversaria.

Matrici attive:
- Impatto / Sforzo;
- Segnale / Rumore;
- Profondita / Urgenza;
- Rischio / Reversibilita;
- Qualita Output.

Output atteso:
- file piu popolati;
- meno template generici;
- esempi concreti;
- struttura leggibile;
- nessun caricamento inutile.

## Regola finale

MD_OS deve essere ricco nei file, ma selettivo nel runtime.

Popolare non significa caricare tutto.
Popolare significa rendere ogni file utile quando viene scelto.
