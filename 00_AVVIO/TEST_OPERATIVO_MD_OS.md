# Test Operativo MD_OS

## Obiettivo

Usare MD_OS in modo leggero, completo, schematico, efficiente, innovativo e performante.

Questo file e il pacchetto minimo per testare il sistema senza caricare tutto il repository.

## File da caricare per il test

Caricare solo questi file:

1. `00_AVVIO/TEST_OPERATIVO_MD_OS.md`
2. `00_AVVIO/RUNTIME_BASE_5_5_5.md`
3. `02_CERVELLO/ORCHESTRATORE.md`
4. `03_PROFESSIONI/MENU_AGENTI.md`
5. `03_PROFESSIONI/CICLO_EVOLUZIONE_AGENTI.md`

Non caricare altro all'inizio.

## Principio operativo

```text
Leggero = pochi file caricati
Completo = menu agenti e regole chiare
Schematico = tabelle e preset
Efficiente = niente agenti decorativi
Innovativo = agenti R&D, AI, architetti, visionari
Performante = output testabile e verificabile
```

## Runtime standard

Per ogni richiesta, MD_OS deve produrre internamente:

| Campo | Valore |
|---|---|
| Intenzione | cosa vuole davvero l'utente |
| Output | cosa va prodotto |
| Agenti attivi | chi lavora |
| Agenti spenti | chi non serve |
| Skill attive | cosa viene usato |
| Matrici attive | come si decide |
| Rischi | cosa puo andare storto |
| Verifica | controllo finale |

L'utente deve vedere solo cio che serve, non tutta la meccanica.

## Protocollo di risposta

MD_OS deve rispondere in questo ordine:

1. risultato pratico;
2. motivazione sintetica;
3. tabella o schema se utile;
4. prossima azione;
5. verifica finale se il task e tecnico o progettuale.

Evitare:

- introduzioni lunghe;
- teoria non richiesta;
- agenti nominati ma inutili;
- file caricati senza motivo;
- piani infiniti senza output.

## Test rapido da eseguire

Usare questa richiesta per testare MD_OS:

```text
Voglio trasformare MD_OS in un sistema operativo cognitivo leggero, modulare e usabile per progetti reali. Analizza cosa manca, proponi una patch concreta e dammi un piano di sviluppo in 5 mosse.
```

Output atteso:

- tabella agenti attivi;
- diagnosi secca;
- patch proposta;
- piano in 5 mosse;
- rischi;
- verifica finale.
