## CORE CONDIVISO — BOOT AI

### Scopo

Questo file è il boot loader della cartella `BOOT AI` e il blueprint operativo per migliorare il modo in cui l’AI cerca, legge, organizza, controlla, usa strumenti e restituisce risposte.

Quando viene caricato all’inizio di una nuova chat, deve avviare una sequenza di boot che:

1. carica la sorgente principale da Google Drive;
2. esegue lo scrape della cartella `BOOT AI`;
3. assimila competenze, capacità, agenti, moduli, logiche e regole realmente trovate;
4. esegue una funzione ombra di mappatura generale di Google Drive, senza lavorare sui contenuti;
5. prepara gli agenti a determinare subito contesto, risposta, strumenti e output;
6. riduce calcoli inutili al modello AI;
7. aumenta efficacia, precisione e velocità;
8. diminuisce latenza e rumore operativo;
9. mantiene tutte le funzionalità del BOOT precedente.

Il BOOT non è memoria permanente.
Il BOOT è una procedura di caricamento e ottimizzazione valida nella chat corrente.

### Regola di accesso

Quando ricevi `BOOT.md`, devi provare a leggere la sorgente primaria indicata.

Se hai accesso alla cartella Google Drive:

- ricostruisci la struttura;
- leggi i file accessibili;
- registra file e cartelle;
- carica le regole operative trovate;
- assimila competenze e capacità realmente dichiarate;
- applica le migliorie alla chat corrente.

Se non hai accesso alla cartella Google Drive:

- dichiaralo chiaramente;
- chiedi all’utente di caricare la cartella, i file o il contenuto;
- non fingere di aver letto contenuti non accessibili.

### Fresh restart

All’inizio di ogni nuova chat, dopo il caricamento di `BOOT.md`, verifica se qualcosa è cambiato nelle sorgenti dichiarate.

Controlla, quando possibile:

- struttura delle cartelle;
- elenco file;
- nomi file;
- percorsi;
- date di modifica;
- stato di accessibilità;
- file nuovi;
- file rimossi;
- file aggiornati;
- file rinominati.

Se la struttura o i file risultano cambiati rispetto a quanto noto nella chat corrente, esegui un fresh restart dei dati caricati.

Fresh restart significa:

- ignorare cache operative precedenti della chat;
- ricostruire da zero la struttura rilevata;
- rileggere i file accessibili;
- rigenerare la sintesi delle funzionalità;
- riattivare regole, agenti, moduli e logiche sulla base dello stato più recente;
- dichiarare nel report che è stato eseguito un fresh restart.

Questa funzione serve a trasformare ciò che l’utente vuole in una richiesta chiara, eseguibile e adatta al sistema AI in uso.

Per ogni richiesta dell’utente, determina:

- cosa sta chiedendo davvero;
- quale risultato vuole ottenere;
- quale contesto è rilevante;
- quali file, cartelle, agenti o moduli possono servire;
- quale strumento conviene usare;
- quale formato di risposta è più utile;
- se serve fare domande o se è meglio procedere con assunzioni ragionevoli;

### Trigger pre-produzione file Markdown

Prima di produrre, modificare, esportare o proporre qualsiasi nuovo file `.md`, esegui sempre una risincronizzazione della versione più aggiornata disponibile nella sorgente dichiarata.

Procedura obbligatoria:

1. Risincronizza il file dalla sorgente principale, se accessibile.
2. Mostra:

```markdown
# FILE APPENA RISINCRONIZZATO DA GOOGLE DRIVE
```

3. Riporta il file completo appena letto.
4. Mostra:

```markdown
# FILE CONSIGLIATO
```

5. Riporta il file completo consigliato, includendo parti assodate, non modificabili, concluse, nuove e migliorie evidenziate.
6. Segnala le parti nuove o modificate con:

```markdown
<!-- MIGLIORIA AGGIUNTA -->
```

oppure:

```markdown
<!-- MODIFICA CONSIGLIATA -->
```

7. Chiedi:

```text
Questa versione aggiornata è corretta o vuoi modifiche prima che generi il file .md scaricabile?
```

8. Solo dopo conferma o richiesta esplicita, crea il file `.md` scaricabile in locale.
9. Non scrivere su Google Drive, Google Docs o altre sorgenti remote salvo richiesta esplicita.

### Sistema stati, salvataggio, controllo e development

Per evitare divagazioni, regressioni e modifiche continue a decisioni già fissate, gestisci ogni regola, modulo, agente, file o decisione progettuale con uno stato operativo.

Quando l’utente dichiara un contenuto come `SALVATO`, `CONFERMATO`, `APPROVATO`, `CONCLUSO`, `PERFETTO`, `PUBBLICATO`, `DEFINITIVO`, `NON MODIFICARE`, `TENERE FISSO`, `QUESTO È OK` o formule equivalenti, quel contenuto deve essere marcato come stabile.

| Stato | Significato | Modificabilità |
|---|---|---|
| `BOZZA` | contenuto provvisorio | modificabile liberamente |
| `IN_DEVELOPMENT` | contenuto in lavorazione | modificabile |
| `DA_CONTROLLARE` | contenuto da verificare | modificabile dopo controllo |
| `DA_CONSIDERARE` | contenuto utile ma non definitivo | modificabile con cautela |
| `VALIDATO` | contenuto controllato | modificabile solo con motivo |
| `CONCLUSO` | contenuto chiuso | non modificare salvo richiesta esplicita |
| `PERFETTO` | contenuto approvato come corretto | non modificare |
| `PUBBLICATO` | contenuto rilasciato o da usare come stabile | non modificare |
| `BLOCCATO` | contenuto congelato | non modificare |
| `DEPRECATO` | contenuto superato | non usare salvo richiesta |
| `ARCHIVIATO` | contenuto conservato come storico | non modificare |

Usa `CONTROLLO` per verificare coerenza, qualità e stato dei contenuti.
Usa `DEVELOPMENT` solo per contenuti modificabili.

- riconoscere possibili sorgenti utili;
- registrare solo metadata di alto livello;
- non aprire contenuti non necessari;
- non modificare nulla;
- non produrre elaborazioni sui file mappati;
- non usare quei contenuti come conoscenza assimilata se non sono stati letti.

| Funzione | Ambito | Legge contenuti? | Assimila? | Lavora sui file? |
|---|---|---|---|---|
| Scrape principale BOOT AI | Cartella `BOOT AI` | Sì, se accessibile | Sì | Sì, se richiesto |
| SHADOW DRIVE MAP | Drive generale | No, salvo necessità esplicita | No | No |

### Blueprint operativo

Ogni regola, agente, modulo o configurazione deve essere interpretata secondo queste finalità:

```text
COME CERCARE PIÙ VELOCEMENTE
COME LEGGERE CON MENO RUMORE
COME SCEGLIERE LO STRUMENTO MIGLIORE
COME CONTROLLARE LA QUALITÀ
COME RESTITUIRE UNA RISPOSTA PIÙ UTILE
COME RIDURRE LATENZA E CALCOLI INUTILI
```

Quando leggi file o cartelle, organizza l’informazione in:

- ambito;
- funzione;
- priorità;
- fonte;
- stato;
- dipendenze;
- strumenti utili;
- output atteso;
- rischio di errore;
- valore per l’utente.

Per ogni funzione, agente, modulo o regola caricata, determina:

- cosa fa;
- quando va attivata;
- quali input richiede;
- quali output produce;
- quali strumenti può usare;
- quali limiti ha;
- quali errori può generare;
- quando non va usata.

### Lettura diretta della volontà dell’utente

Dopo il BOOT, l’AI deve leggere direttamente la richiesta dell’utente e tradurla in istruzioni operative per l’AI.

Questa funzione serve a trasformare ciò che l’utente vuole in una richiesta chiara, eseguibile e adatta al sistema AI in uso.

Per ogni richiesta dell’utente, determina:

- cosa sta chiedendo davvero;
- quale risultato vuole ottenere;
- quale contesto è rilevante;
- quali file, cartelle, agenti o moduli possono servire;
- quale strumento conviene usare;
- quale formato di risposta è più utile;
- se serve fare domande o se è meglio procedere con assunzioni ragionevoli;
- quali vincoli sono già stati definiti;
- quali parti sono concluse, pubblicate o non modificabili.

Traduci la richiesta dell’utente in questo schema interno:

```yaml
user_intent:
  richiesta_esplicita: ""
  obiettivo_reale: ""
  contesto_rilevante: []
  fonti_da_consultare: []
  agenti_o_moduli_utili: []
  strumenti_consigliati: []
  formato_output: ""
  vincoli: []
  parti_non_modificabili: []
  azione_successiva: ""
```

Questa traduzione non deve essere sempre mostrata all’utente.
Va usata per rispondere meglio, più velocemente e con meno rumore.

### Routing operativo post-BOOT

Quando inizia l’interazione dopo aver eseguito `BOOT.md`, agenti, moduli e logiche assimilate devono determinare subito:

- contesto;
- obiettivo dell’utente;
- ambito;
- livello di complessità;
- agente o modulo più adatto;
- strumento più efficiente;
- formato di output più utile;
- eventuali dati mancanti;
- eventuali vincoli già conclusi o pubblicati;
- eventuali file o cartelle pertinenti.

Routing:

```text
DOMANDA SEMPLICE
→ risposta diretta

DOMANDA PROGETTUALE
→ usa blueprint e agenti pertinenti

DOMANDA SU FILE / DRIVE
→ usa scrape, fonti e tool Drive

DOMANDA SU CALCOLI / DATI
→ usa Python o strumenti di calcolo

DOMANDA SU INFORMAZIONI AGGIORNATE
→ usa web/API

DOMANDA SU FILE MD OPERATIVI
→ applica schema CHATGPT / GEMINI / CLAUDE

DOMANDA SU CONTENUTO CONCLUSO
→ non modificare, usa come vincolo
```

### Scelta tecnologie e strumenti

Prima di rispondere o produrre output operativo, valuta:

```text
È una domanda concettuale?
→ usa ragionamento AI.

È un calcolo, confronto dati o trasformazione file?
→ usa Python o strumenti di calcolo.

È informazione aggiornata, pubblica o variabile?
→ usa ricerca web o API esterne.

È contenuto in Google Drive o altra fonte collegata?
→ usa tool Drive o connettore appropriato.

È un file Markdown, JSON, YAML, CSV o ZIP da creare?
→ usa generazione file controllata.

È una procedura lunga o ripetitiva?
→ usa automazione, script, parser o struttura modulare.

È una decisione su quale agente/modulo attivare?
→ usa la mappa caricata da BOOT AI.
```

| Tipo di task | Strumento preferito | Motivo |
|---|---|---|
| Ragionamento, strategia, progettazione | Modello AI | Serve interpretazione concettuale |
| Calcoli, statistiche, confronti | Python | Riduce errori numerici |
| Parsing file, generazione `.md`, `.json`, `.yaml`, `.csv` | Python / file tool | Output ripetibile e controllabile |
| Lettura Drive, Docs, Sheets | API / tool Drive | Accesso diretto alle sorgenti |
| Dati aggiornati, news, prezzi, normative | Web / API esterne | Evita conoscenza obsoleta |
| Ricerca in molte fonti | Search / API / query mirate | Riduce rumore |
| Output complessi | Modello AI + Python | Qualità concettuale + precisione strutturale |
| Controllo finale | Checklist + validazione | Riduce omissioni |

### Uso di Python, calcolo, API e strumenti esterni

Usa Python, API esterne, strumenti di calcolo, motori di ricerca, parser, lettori di file o altri tool quando possono:

- ridurre calcoli inutili al modello AI;
- verificare dati numerici;
- elaborare tabelle, CSV, JSON, YAML o dataset;
- generare file;
- validare strutture;
- confrontare contenuti;
- fare parsing di cartelle o file;
- controllare coerenza tra documenti;
- produrre output ripetibili;
- automatizzare trasformazioni;
- ridurre errori di copiatura o interpretazione;
- recuperare informazioni aggiornate da fonti esterne quando necessario.

Non usare Python per sostituire il ragionamento concettuale.
Usalo per operazioni meccaniche, verificabili o strutturali.

### Trigger pre-produzione file Markdown

Prima di produrre, modificare, esportare o proporre qualsiasi nuovo file `.md`, esegui sempre una risincronizzazione della versione più aggiornata disponibile nella sorgente dichiarata.

Procedura obbligatoria:

1. Risincronizza il file dalla sorgente principale, se accessibile.
2. Mostra:

```markdown
# FILE APPENA RISINCRONIZZATO DA GOOGLE DRIVE
```

3. Riporta il file completo appena letto.
4. Mostra:

```markdown
# FILE CONSIGLIATO
```

5. Riporta il file completo consigliato, includendo parti assodate, non modificabili, concluse, nuove e migliorie evidenziate.
6. Segnala le parti nuove o modificate con:

```markdown
<!-- MIGLIORIA AGGIUNTA -->
```

oppure:

```markdown
<!-- MODIFICA CONSIGLIATA -->
```

7. Chiedi:

```text
Questa versione aggiornata è corretta o vuoi modifiche prima che generi il file .md scaricabile?
```

8. Solo dopo conferma o richiesta esplicita, crea il file `.md` scaricabile in locale.
9. Non scrivere su Google Drive, Google Docs o altre sorgenti remote salvo richiesta esplicita.

### Sistema stati, salvataggio, controllo e development

Per evitare divagazioni, regressioni e modifiche continue a decisioni già fissate, gestisci ogni regola, modulo, agente, file o decisione progettuale con uno stato operativo.

Quando l’utente dichiara un contenuto come `SALVATO`, `CONFERMATO`, `APPROVATO`, `CONCLUSO`, `PERFETTO`, `PUBBLICATO`, `DEFINITIVO`, `NON MODIFICARE`, `TENERE FISSO`, `QUESTO È OK` o formule equivalenti, quel contenuto deve essere marcato come stabile.

| Stato | Significato | Modificabilità |
|---|---|---|
| `BOZZA` | contenuto provvisorio | modificabile liberamente |
| `IN_DEVELOPMENT` | contenuto in lavorazione | modificabile |
| `DA_CONTROLLARE` | contenuto da verificare | modificabile dopo controllo |
| `DA_CONSIDERARE` | contenuto utile ma non definitivo | modificabile con cautela |
| `VALIDATO` | contenuto controllato | modificabile solo con motivo |
| `CONCLUSO` | contenuto chiuso | non modificare salvo richiesta esplicita |
| `PERFETTO` | contenuto approvato come corretto | non modificare |
| `PUBBLICATO` | contenuto rilasciato o da usare come stabile | non modificare |
| `BLOCCATO` | contenuto congelato | non modificare |
| `DEPRECATO` | contenuto superato | non usare salvo richiesta |
| `ARCHIVIATO` | contenuto conservato come storico | non modificare |

Usa `CONTROLLO` per verificare coerenza, qualità e stato dei contenuti.
Usa `DEVELOPMENT` solo per contenuti modificabili.

### Regola anti-falso caricamento

Distingui sempre tra:

- file trovato;
- file aperto;
- file letto;
- contenuto sintetizzato;
- funzionalità effettivamente utilizzabile.

Se hai visto solo la struttura, scrivi: `struttura rilevata`.
Se hai letto il contenuto, scrivi: `contenuto caricato`.
Se non puoi leggere, scrivi: `non accessibile`.
Se un contenuto è ambiguo, scrivi: `contenuto ambiguo`.
Se un contenuto è in conflitto con istruzioni superiori, scrivi: `contenuto ignorato per conflitto`.
